Phpunit的一些看法

作者：迷茫者 发布于：2015-3-31 6:00 Tuesday 分类：有道云笔记

Phpunit 的一些看法

作者： 王心怡

 

     总觉得最近应该写点啥，哈哈， 不知道是聊聊自己最近学习的 Common Lisp ， 还是讲讲 CCoffeeOffice呢~~ , 不过这些都刚学， 发出来怕被各种拍砖，而且又属于冷门的东西，既然这两年在敲PHP，就聊一下PHP相关的工具吧， 思来想去， Composer ?，phpcs(codesniffer) ?，ApiGen|Phpdocument?，phing?， 好吧~~ 最近公司在要求使用phpunit，今天还是讲讲PhpUnit 这个工具吧。

 

     这里我不会讲神马PHPUnit 和其他单元测试工具之类的关系，单元测试工具， 说白了就是一种自动测试工具包，任何程序语言都有（很多还不知一种），就连javascript都有jasmine, YUI Tester 这样的单元测试工具。  说到单元测试， 实际就不能不说TDD(测试驱动开发），我一直觉得如果单元测试脱离了TDD, 真心没啥好说的了。 所以说是介绍phpunit 的实践，但是这里还是以介绍TDD为主， 使用TDD的开发方式会经历一阵痛苦，挑战习惯的过程，不知道最后有几个人能最后坚持以TDD为准则， 但是相信我，经过大量练习和实践后，你会发现这太酷了， 并且对于以后整个项目使用CI系统提高软件质量和降低风险有极为重要的作用！

 

一 我的单元测试初体验及我所理解的单元测试

     先讲一个我的经历： 大概在2010年初的时候， 我在一个电商网站任职， 当时有个任务是做网站上一个推荐系统的接口， 推荐系统接口功能要求有几点： 1：根据用户搜索词找出最适合的产品，比如用户搜 “水” ，搜索结果就应该是“矿泉水”在最前面，而不应该是”补水面膜“，”水果“这样的结果， 用户搜索”肥皂”，就应该出来“香皂”这样的结果，而不应该没有查询结果 2. 把这些适配词，相近词，网站词频高的词提取出来， 发送到推荐系统接口 3，把用户浏览过的商品记录，购物车商品， 用户购买历史发送到推荐系统接口， 4.最后从推荐系统获取相关商品id.。 当时发现一个问题就是：从编码实现某个功能，到我测试运行，周期是如此之长~  要完成一个整个的周期测试，我需要 ‘注册用户’ - ‘浏览几个特定的网页’ - ‘放几个制定的商品到购物车里’ - ‘购买几件产品产生购买记录’ - 测试结果， 当结果不符，我再修改代码， 再经历一次上述过程，周而复始， 整个过程是如此痛苦，甚至改一点点推荐的参数， 稍微改一下不同的推荐权重，我都需要经过上面一串的步骤， 我发现了，这种过程，简直又傻又笨啊！ 而且一点对代码执行结果的安全感也没有！

 

   后来， 我使用了一种方法，

  1. 把上面的那些功能点拆分成不同的单元(unit) : 每个单元实现一个单独功能， 单元要求是： 接受对应的数据输入， 返回相应的数据输出。 于是我根据这个拆分为 “根据某个词返回相近词”， “对某个单字返回最合适的词（根据词频和分类标签)” ，”通过用户id获取浏览商品记录” “获取当前购物车”  “直接把某个商品放入某个用户id浏览历史记录里” ,”设置购买记录” “获取购买记录” “发送数据到推荐系统接口“ 等单元。  

 

  1. 编写数据提供器， 我的数据提供器是各种数组（比如： 用来测试的用户id的数组， 对应用户id 需要放入浏览记录的商品id 数组, 需要放入购物车的商品id 数组,  放入历史购买记录的商品id 数组 等等数据）
  2. 编写测试代码， 依次迭代上面的数据提供器，根据数据提供器的描述， 对测试数据进行声称， 测试代码通过这些数据调用上面的 功能单元， 判断返回结果。 当然， 当时我没有使用任何单元测试工具提供的断言工具， 而只是直接输出返回数据， 但是已经让我的工作效率提升了很多。 而且对每一个unit 的结果都能做到心中有数。 并且在以后发现问题时候， 通过修改数据提供器， 执行测试后能快速定位到哪一个unit 出现的问题。 让我受益匪浅~

 

诚然，当时我并不知道这就是单元测试(unittest)的雏形， 但是这个过程，和我理解的单元测试是一致的：

 1）缩短编码和运行周期， 使用单元测试， 我能保持我代码保持能立即执行的节奏，比如实现“对某个单字返回最合适的词” 这个单元时， 我就可以切开两个窗口(vim里可以ctrl+w, s) ,左边窗口写我这个单元的代码， 右边窗口写对应这个单元的测试，再开一个shell 执行这个单元测试，并维持30-60秒执行一次的节奏， 检查这个单元代码的执行情况， 并最后直到单元测试通过的标准。 而我们传统的编写方式，可能需要在编写几百行代码后才执行一次，而且编写到执行的周期如此之长，步骤之多是可以想象的。

 2) 保持结果的确定性, 通过每个单元的单元测试，通过单元测试里的断言代码(assertEquals AssertTrue) , 我们就可以保证，通过我们给单元功能的一些数据执行后，返回的数据是我们确定正确的， 这样，我们在以后对代码进行任何修改的时候， 如果要检查是否对之前功能进行了破坏， 只要再次执行所有单元测试代码即可进行检查（呵呵，这就到了CI 持续集成的范畴了)

 3)  增强了对自己代码的信心： 当你的单元测试都通过了，你可以自信的说句， 我这里代码没问题.(额, 大概吧，实际上，自己写的单元测试也很难避免一些坑。。。）

 4)  单元测试也可以作为文档使用， 当你写好了代码的单元测试， 有同事需要调用你的接口，直接跟他说“你看我单元测试吧”。

 

我认为对单元测试的一些误区：

  1. 片面强调代码覆盖率， 有个同事喜欢不断强调什么代码覆盖，我很想问他凭什么，如果坚持测试代码驱动， 那么根据测试代码写得生产代码，在这种原则下说代码覆盖有意义么？ 
  2. 单元测试不会影响开发效率。“稍后再写测试”实际工作中是不可能的， 代码完成后写测试，仔细看来，这种测试可以达到很高的覆盖率，是的，但是你的测试代码是根据你产品代码产生的，没有很高的覆盖率才奇怪了呢。。。但是这种代码后的测试只是一种防守。 而先行写的测试则是进攻， 事后写的测试已经受制于你的代码， 而代码又是你写成的，你主观上已经规避了某些问题。

 

二 TDD行为准则

  1. 在编写好某个功能单元（unit)的单元测试(失败的单元测试前） 不要写任何产品代码。

从TDD(测试驱动开发）这个词就可以看出， 整个开发流程是以测试代码驱动的， 它最主要的核心是测试现行， 先写某一个unit 的测试代码，然后通过这个测试代码为基础，实现生产代码。 比如我上面那个“获取某个用户购物车数据” ，我们就可以先写如下测试代码：

   Class CartTest extends PHPUnit_Frameword_TestCase

   {

       Public testGetCart() {

           $userid = 123;

           $cart = new Cart();

           $data = $cart->getCarts($usrid);

           $this->assertNotEmpty($data);

           $this->assertTrue(is_array($data));

       }  

   }

而这时，我们的生产代码还没有实现，这时我们的生产代码是这样:

Class  Cart

{

Public function getCarts($userId) 

{

    // 没实现

}

}

 

所以，这个时候我们执行单元测试时候，会直接显示 F(表示我们这个单元测试结果失败)  这时正常的， 这就需要我们下一步去实现我们的单元测试，

Public function getCart($userId) {

// 通过$userId 获取购物车内容

return $carts;

}

这样，执行phpunit 单元测试， 代码执行通过（控制台会显示一个 . 表示通过), 这样我们就可以继续下一个单元的开发了。

 

  1. 只要有一个单元测试失败了， 就不要写单元测试代码。

很多同学遵守先行测试的TDD准则， 但是会有个情况发生，比如这种代码:

Class CartTest extends PHPUnit_Frameword_TestCase

   {

       Public testGetCart() {

           $userid = 123;

           $cart = new Cart();

           $data = $cart->getCarts($usrid);

           $this->assertNotEmpty($data);

           $this->assertTrue(is_array($data));

       }  

       Public testAddCart()

       {

          $cartData = array(....);

          $cart = new Cart();

          $this->assertTrue($cart->addCart($cartData));

       }

       /// 等等，单元测试代码

   }

但是在生产代码里， 所有的方法都没实现。。。

这样执行单元测试， 全是一堆F. 

这样会有个问题，就是容易造成不知道从何入手开始完成这些测试代码。 造成各种混乱。 所以，应该是  测试代码-》 生产代码 -》测试代码-》生产代码不断循环， 两套代码互相增长，互相补充。 两者相辅相成。

 

  1. 产品代码恰好能够让当前失败的单元测试成功通过即可， 不要多写。 一个单元以实现一个功能的unit 为主， 这就需要我们在一开始设计时隔离出待测试的代码功能， 为了编写测试，我们首先考虑吧单个unit 解耦， 如果在一个生产函数里实现了过多“附加功能”，减少了代码覆盖率是一定的，而且因为不是单元测试驱动要实现的， 或让代码耦合度加大，最后变成最终产品无法测试的情况。

 

三．练习

事实上， TDD的方法对于我们来说，开始很难习惯，行为规范很难调整过来， 所以一些国外的程序员团队流行平时做一些练习， 使用TDD的方法，写一些小功能，在不断练习中达到纯熟，并且有利于训练问题与解决方案之间联系。 在<then clean coder >这本书里提出的一个叫做卡塔的活动很有意思,Rob写到，

“2005年我出席在英国谢菲尔德举办的xp2005大会，会上我参加了”编程柔道场”的主题活动，跟他们一起用测试驱动开发来编写Conway’s Game of life ， 他们称其为“卡塔””

 

于是我把卡塔习题下载了下来， 并用php改写了下， 很有意思， 到时候我放到网盘共享下。

 

四． 懒得写了

    就写这么多吧， phpunit 具体使用很简单， 手册也说的很清楚，或者照着别人的单元测试写一下也行，我这里主要说的是单元测试，没有覆盖功能测试， 功能测试在对于我们web开发实现起来很麻烦， 我以前使用过SeleniumServer 实现web的功能测试。 但是后来还是比较难用， 最近考虑使用php实现一个brower client 模拟， 实际symfony2 就实现了个这么个东东， 看了下，里面一个kernel 就几万行代码了， 超级复杂，symfony2 的解耦很好，实际直接拿过来就行了， 可惜是php5.4实现的，，namespace, static:: use闭包等功能目前php5.2铁定不能用， 而我目前没有改造计划的想法。。。

 

 

另： phpunit --coverage-html ./Tesclass 测试文件 可以出代码覆盖率报告， 但是我总觉得， 如果你真的按照TDD的准测开发， 你的代码根据测试进行驱动的， 那么你的代码覆盖率就是高（因为代码就是为了实现测试写的），除非你写了很多测试功能要求以外的东西。 这个代码覆盖率，个人觉得也就那种写好代码后写测试的才会考虑的吧。
